CRITICAL BUGS (must fix before any production use)
BUG 1 — getNextNfeNumero() is non-atomic (race condition → duplicate numbers)

storage.ts:2339 does:

SELECT COALESCE(MAX(numero::integer), 0) FROM nfe_emissoes
then adds +1 in application code and inserts. Under concurrent emissão (emitir-lote batches 5 items in Promise.all, cron batches up to 500), two processes race and both read the same MAX. Both attempt to insert the same numero. The unique index on (serie, numero) catches this and throws a DB error — but it means the second NF-e fails completely. This is a hard blocker for any multi-order batch in production.

Fix options:

Correct: Use a PostgreSQL sequence — CREATE SEQUENCE nfe_numero_seq; DEFAULT nextval('nfe_numero_seq') on numero
Acceptable: Wrap in a pg_advisory_xact_lock around a transaction: lock → read MAX → insert → release
BUG 2 — UF hardcoded to 'SP' in production send path

routes.ts:2709:

const uf = nfe.ambienteFiscal === 'producao' ? 'SP' : 'SP';
Both branches return 'SP'. For any emitter in MG, RJ, RS, PR, SC, or any other UF, the NF-e is sent to SP's SEFAZ URLs — which will reject it. The URL map in nfeUrl.ts already knows all UFs; it just never receives the right one. Must read cfg.uf or cfg.addressState from companyConfig.

IMPORTANT GAPS (features missing for complete SEFAZ compliance)
GAP 1 — Cancel NF-e sends nothing to SEFAZ

DELETE /api/nfe/:id only does storage.updateNfeEmissao(id, { status: 'cancelada' }). The enviarCancelamentoNFe() function already exists in nfeSender.ts but is never called from the cancel route. SEFAZ requires a formal CAN-NF-e event within 24h of authorization for authorized notes. Without it, the SEFAZ still considers the NF-e valid — the cancellation only exists locally.

GAP 2 — CC-e sends nothing to SEFAZ

POST /api/nfe/:id/cce creates a DB record and passes all business rules (limit, time window, status, motivo length) but never generates or transmits a CC-e XML to SEFAZ. The note is correct at SEFAZ until the CC-e is transmitted.

GAP 3 — No NF-e inutilização endpoint

SEFAZ requires inutilização when a number range is skipped without being used. There is no endpoint for this.

GAP 4 — nfe-transmit.service.ts production path throws immediately

Even if NFE_SEFAZ_MODE=production is set, the transmit service throws NFE_SEFAZ_PRODUCTION_NOT_IMPLEMENTED. The only working production path is the legacy one gated by CERT_PATH + CERT_PASSWORD env vars — which has BUG 2 above.

GAP 5 — DANFE reflects current order state, not emission-time state

GET /api/nfe/:id/danfe calls buildNFeInput(orderId) at download time rather than using the stored xmlGerado / xmlAutorizado. If the order is modified after emission, the DANFE will not match the transmitted XML. The correct approach is to parse the stored authorized XML, or render the DANFE from the stored XML content.

MODERATE ISSUES
M1 — nfeEmissoes has no tenantId column

Tenant scope is applied via subquery JOIN through orders.companyId. NF-es with orderId = NULL (if any are ever created directly) have no tenant scope and fall through to all tenants. Low risk today since all current paths require an orderId, but creates a subtle invariant to maintain.

M2 — Fiscal status transition to nota_liberada is implicit

The cron and /api/nfe/eligible both require orders.fiscal_status = 'nota_liberada'. The schema comment only documents nota_pendente | nota_exportada | nota_emitida | nota_cancelada — nota_liberada is missing from the comment. The actual transition that sets nota_liberada needs to be explicitly identified and documented to avoid it becoming an invisible manual step.

M3 — Two production send paths with inconsistent logic

Path A: NFE_SEFAZ_MODE != 'production' → transmitirNFe() → mock (or throws in production)
Path B: CERT_PATH + CERT_PASSWORD set → direct enviarNFeSEFAZ() → real SEFAZ (but UF='SP' bug)
These two paths were built at different times and are not unified. When real production is wired, one path should be the canonical implementation.

M4 — cStat column missing in some status update paths

The reenviar and corrigir-reenviar routes call storage.createNfeEmissao() to create a new NF-e but never write back cStat/xMotivo for the old rejected NF-e. The old record stays with whatever cStat it had at rejection, which is correct, but the new record starts with no cStat — fine until SEFAZ responds.
