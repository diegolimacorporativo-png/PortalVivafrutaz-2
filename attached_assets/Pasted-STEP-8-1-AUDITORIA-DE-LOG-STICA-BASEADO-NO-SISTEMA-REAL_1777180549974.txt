## STEP 1 — DESCOBERTA DE ESTRUTURA

Verificar no backend:

### PEDIDOS

Campos importantes:

- status  
- workflowStatus  
- deliveryDate  
- companyId  

👉 Esses já são a base da logística  

---

### WORKFLOW (ESSENCIAL)

Você já tem:

CREATED  
→ PENDING_APPROVAL  
→ APPROVED  
→ PROCESSING  
→ READY  
→ INVOICED  
→ SHIPPED  
→ DELIVERED  

👉 LOGÍSTICA começa aqui:

- READY → pronto para roteirização  
- SHIPPED → saiu para entrega  

---

### LOGS (já existentes)

Tabela: system_logs  

👉 já registra:

- transições  
- ações  
- usuário  

👉 isso será a base da timeline logística  

---

### VERIFICAR SE JÁ EXISTE

Buscar no código:

- route  
- delivery  
- driver  
- vehicle  
- map  
- lat  
- lng  

👉 mapear se algo disso já existe

## STEP 2 — IDENTIFICAR GAPS (SEM INVENTAR)

Responder:

❓ Existe vínculo pedido → motorista?  
- ❌ não existe → GAP  

❓ Existe agrupamento de pedidos (rota)?  
- ❌ não existe → GAP  

❓ Existe controle de status de entrega detalhado?  
Hoje só tem:

- SHIPPED  
- DELIVERED  

👉 falta granularidade  

❓ Existe painel de motorista?  
- ❌ não existe → GAP

## STEP 3 — ONDE A LOGÍSTICA ENTRA

👉 Integração correta com seu sistema atual:

📌 READY  
- pedido pronto  
- entra na fila de roteirização  

📌 SHIPPED  
- motorista atribuído  
- rota definida  
- pedido sai para entrega  

📌 DELIVERED  
- finalização  

👉 NÃO criar novos estados ainda  

---

## STEP 4 — USUÁRIO MOTORISTA

Base existente:

- tabela users  
- campo role  

👉 criar role: DRIVER  

Regras:

✔ motorista vê apenas pedidos atribuídos  
✔ NÃO pode editar pedido  
✔ NÃO acessa admin  

👉 isso é controle de acesso, não sistema separado
