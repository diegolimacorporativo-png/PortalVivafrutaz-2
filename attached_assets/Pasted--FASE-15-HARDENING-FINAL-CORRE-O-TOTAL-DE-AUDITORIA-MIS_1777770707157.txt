# 1. AUTH / SESSION — CORREÇÕES OBRIGATÓRIAS

## 1.1 SESSION FAIL-CLOSED (CRÍTICO)
- authCoreService.validateSession NÃO pode retornar true em erro de DB
- QUALQUER erro → REJEITAR sessão
- Remover completamente comportamento fail-open

---

## 1.2 DEVICE BINDING (CRÍTICO)
- X-Device-Id deve ser OBRIGATÓRIO para sessão válida
- Remover qualquer bypass por ausência de header
- Remover exceção srv-* se permitir bypass de segurança
- Sessão sem deviceId = INVALID

---

## 1.3 TOKEN VERSION CONSISTENCY
- Sessões sem tokenVersion devem ser consideradas INVÁLIDAS
- Remover qualquer bypass de compatibilidade silenciosa
- revokeAllSessions deve invalidar TODAS sessões antigas corretamente

---

## 1.4 MIDDLEWARE DUPLICADO (CRÍTICO)
Unificar em uma única fonte de verdade:

- requireAuth (core/http)
- auth middleware (middleware/auth)

REGRAS:
- Apenas UM middleware ativo no sistema
- Remover divergência entre:
  - session.role vs session.userRole
  - session.userId vs session.id
- Garantir execução única e consistente

---

## 1.5 LOGIN FLOW CONSISTENCY
- Admin e Company DEVEM usar o MESMO _runAuthFlow
- Remover qualquer lógica paralela fora do engine
- maintenanceMode e mustChangePassword devem ser flags internas do fluxo, não branches separados

---

## 1.6 LOGGING CLEANUP (CRÍTICO)
Remover completamente de auth/session/NF-e:

- console.log
- console.warn
- console.error

Substituir SOMENTE por:
- logAuthEvent
- logSecurityEvent

Aplicar em:
- auth.service
- auth.controller
- session guard
- security blocker
- NF-e transmit flow

---

# 2. NF-e — CORREÇÕES DE IDPOTÊNCIA

## 2.1 ESTADO "ENVIANDO"
- Garantir que status "enviando" não pode gerar dupla emissão após crash
- Ajustar lógica para ser idempotente no guard
- status procedural NÃO pode permitir reprocessamento duplicado

---

## 2.2 CONSISTÊNCIA DE TRANSAÇÃO
- Nenhuma NF-e pode ser emitida duas vezes por retry automático
- Garantir proteção via:
  - advisory lock existente
  - status guard reforçado
