# 🧠 PROBLEMA CENTRAL

O sistema tem 1 inconsistência crítica:

## ❗ ADMIN LOGIN está fora do modelo de segurança unificado

Enquanto COMPANY login usa:

- L1 rate limit (memory)
- L2 rate limit (DB)
- auth_attempts logging
- AuthCoreService logging

👉 ADMIN login usa apenas:
- IP limiter + lockout simples

---

# 🔧 IMPLEMENTAÇÃO — PASSO 1 (CRÍTICO)

## 🔐 ALINHAR ADMIN LOGIN COM COMPANY LOGIN

📍 arquivo: auth.service.ts

### ANTES (simplificado)
- loginIpLimiter (IP only)
- bcrypt verify
- loginAttempts + lockout
- sem AuthCore

---

### DEPOIS (CONSOLIDADO)

Adicionar fluxo equivalente ao COMPANY:

ts id="admin_rate_l1" await checkUserRateLimit(admin.id); 

ts id="admin_rate_l2" await authCoreService.checkDbRateLimit({   userId: admin.id,   ip,   endpoint: "admin_login" }); 

---

### LOG DE TENTATIVA (OBRIGATÓRIO)

ts id="admin_attempt_log" await authCoreService.recordAttempt({   userId: admin.id,   companyId: null,   ip,   endpoint: "admin_login",   success: false }); 

---

### LOG DE SUCESSO

ts id="admin_success_log" authCoreService.logAuthEvent(   AUTH_EVENTS.LOGIN_SUCCESS,   {     userId: admin.id,     ip,     endpoint: "admin_login"   } );

# 🔧 IMPLEMENTAÇÃO — PASSO 3

## 📊 UNIFICAR LOGGING PIPELINE

### PROBLEMA ATUAL

- logSecurity() → alertEngine + console
- logAuthEvent() → buffer circular

👉 dois pipelines diferentes

---

### SOLUÇÃO

📍 dentro de AuthCoreService

Criar wrapper único:

ts id="unified_logger" logAuthEvent(event, payload) {   // 1. buffer interno (observability)   logSecurityEvent(event, payload);    // 2. alert engine apenas para críticos   if (isCritical(event)) {     logSecurity(`[SECURITY] ${event}`, payload);   } } 

---

### RESULTADO

- 1 chamada
- 2 outputs consistentes
- sem duplicação externa

---

# 🔧 IMPLEMENTAÇÃO — PASSO 4

## 🧹 REMOVER ACOPLAMENTO sessionGuard → storage

### PROBLEMA

sessionGuard chama storage diretamente para logs.

---

### SOLUÇÃO

ts id="guard_fix_before" // ❌ sessionGuard usa storage storage.createLog(...)
