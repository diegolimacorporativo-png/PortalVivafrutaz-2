FASE: PASSWORD RECOVERY — IMPLEMENTAÇÃO SEGURA (DEV → PRODUÇÃO)

OBJETIVO:
Implementar recuperação de senha funcional, segura e pronta para produção, SEM quebrar o sistema atual.

IMPORTANTE:
- NÃO alterar auth existente
- NÃO alterar fetchWithAuth
- NÃO alterar multi-tenant
- IMPLEMENTAÇÃO incremental

========================================================
FASE 1 — TOKEN (OBRIGATÓRIO)
========================================================

Criar estrutura de reset:

Tabela: password_reset_tokens

Campos:
- id
- userId
- token (string única)
- expiresAt (Date)
- createdAt

========================================================
FASE 2 — FORGOT PASSWORD
========================================================

Rota:
POST /api/auth/forgot-password

Comportamento:

✔ receber email
✔ SEMPRE retornar 200 (não revelar se existe)

Se usuário existir:
- gerar token seguro (crypto.randomUUID ou similar)
- salvar no banco com expiração (15 minutos)

MODO DEV:
- NÃO enviar email
- logar no console:

RESET LINK:
http://localhost:3000/reset-password?token=XYZ

========================================================
FASE 3 — RESET PASSWORD
========================================================

Rota:
POST /api/auth/reset-password

Entrada:
- token
- novaSenha

Validação:

✔ token existe
✔ não expirado
✔ pertence a usuário válido

Execução:

✔ hash da nova senha
✔ atualizar usuário
✔ deletar token
✔ remover bloqueio (se existir)

========================================================
FASE 4 — FRONTEND
========================================================

Criar página:
/reset-password

Fluxo:

- ler token da URL
- permitir inserir nova senha
- chamar API

Estados:

✔ token inválido → erro claro  
✔ token expirado → erro claro  
✔ sucesso → redirecionar login  

========================================================
FASE 5 — SEGURANÇA
========================================================

✔ token uso único  
✔ expiração obrigatória  
✔ senha sempre hash  
✔ não expor existência de email
